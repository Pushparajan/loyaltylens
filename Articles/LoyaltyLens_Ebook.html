Section 1

Architecture Overview

LoyaltyLens is a full-stack, production-grade loyalty platform built as ten independently deployable microservices. Each service maps to one of the 15 Claude Code prompts — build them incrementally and compose the system progressively.

S1

Customer Service

Customer profiles, segmentation data, identity management, event ingestion

S2

Offer Engine

Offer creation, eligibility rules engine, budget enforcement, redemption tracking

S3

Segmentation Engine

Dynamic customer grouping, rule-based + ML segmentation, real-time qualification

S4

Campaign Orchestration

Multi-step journeys, trigger-based workflows, state machine, channel coordination

S5

Personalization Engine

Next-best-action, real-time ranking, rule + model-based decisions

S6

LLMOps Pipeline

Prompt versioning, eval harness, drift monitor, audit logging, CI/CD for ML

S7

RLHF Feedback Loop

Event-driven feedback capture, preference dataset, retraining trigger, RLHF fine-tuning

S8

Analytics Service

KPI tracking, funnel analysis, cohort analysis, aggregation pipelines

S9

Security & RBAC

JWT authentication, RBAC + ABAC middleware, policy enforcement, audit logging

S10

Infrastructure / IaC

Terraform (ECS/EKS, RDS, MSK, API Gateway), CI/CD pipeline, multi-tenant deployment

Data Flow

Customer Events → Feature Pipeline (Pandas/DuckDB) → Feature Store → Propensity Scorer (PyTorch) → RAG Retrieval (LangChain + pgvector) → LLM Copy Generator → Personalization Engine → Offer Engine → Campaign Orchestration → LLMOps → RLHF Feedback → Kafka → Analytics

Enterprise Deployment Architecture

LoyaltyLens operates as a private SaaS platform: the runtime executes entirely within the client's cloud boundary, while a vendor-owned control plane handles licensing, updates, AI governance, and observability metadata.

LayerOwnershipResponsibilities
Control Plane Vendor Tenant provisioning, licensing, feature flags, release orchestration, AI model registry & governance, observability metadata, JIT support access
Runtime Plane Client VPC All PII and transaction data processing, API gateway, loyalty orchestration, offer engine, points ledger, profile service, AI personalization inference, operational databases, audit logging
Positioning Statement

"Private SaaS loyalty platform with AI-driven personalisation, deployed in your cloud with continuous innovation and zero data compromise."

Security Architecture

  • AES-256 at rest

    All operational and ledger databases with BYOK (Bring Your Own Key)

  • mTLS in transit

    TLS 1.3 minimum on all inter-service and client-to-API communication

  • Zero-trust network

    All service-to-service calls authenticated even within the VPC

  • RBAC + ABAC

    Role-based + attribute-based policies for fine-grained data access

AI Governance Modes

Federated LearningModel gradients shared across tenants; raw data never moves; patterns improve globallyOpt-in

What to build

    Data Generator

  • 2

  • 3

    DuckDB Feature Store

    Versioned tables: features_v1, features_v2 — enabling feature lineage tracking

  • 4

    FastAPI Serving Endpoint

    GET /features/{customer_id} — returns latest feature vector in JSON

Section 5 · S2

Offer Engine — Propensity Scoring

A PyTorch tabular transformer (TabTransformer-lite) trained on feature store output to predict offer redemption probability, with MLflow tracking and a SageMaker stretch goal.

Model Architecture

Input (6 features) → Linear(6→64) + LayerNorm → TransformerEncoder (2 layers, d_model=64, nhead=4) → MLP: Linear(64→32) → ReLU → Dropout(0.1) → Linear(32→1) → Sigmoid → propensity_score [0, 1]

Section 7 · S4

Campaign Orchestration — LLM Copy Generator

A prompt template registry (YAML) with versioned prompts, LLM backends (HuggingFace / OpenAI-compatible), and a CLIP-based brand image scorer for GenAI content alignment.

Prompt Version Control

Each YAML file is the source of truth — commit all prompt changes to Git. prompt_version is logged with every inference record (critical for drift analysis). Never hardcode prompts inline — always load from the registry at runtime.

Output OfferCopy dataclass

@dataclass
class OfferCopy:
    headline: str        # e.g. "Earn double stars this weekend!"
    body: str            # max 40 words
    cta: str             # "Redeem now" / "Shop today"
    tone: str            # friendly | urgent | exclusive
    model_version: str
    prompt_version: str
    latency_ms: float
    token_count: int
Section 8 · S5

LLMOps Pipeline — Eval, Drift & CI/CD

Production LLMOps with prompt versioning registry, automated eval harness (BLEU + ROUGE + LLM-as-judge), PSI drift monitoring, and a GitHub Actions CI/CD pipeline with eval gate.

Eval Score Composition

aggregate_score = (
  0.20 * bleu_score          # lexical overlap
  + 0.20 * rouge_l_score     # recall-oriented overlap
  + 0.20 * coherence / 5     # LLM judge: logical flow
  + 0.20 * brand_alignment / 5  # LLM judge: brand voice
  + 0.20 * cta_strength / 5  # LLM judge: call-to-action
)
# CI gate: blocks merge if aggregate_score < 0.75

PSI Drift Monitor

PSI ValueStatusRecommended Action
< 0.1OK — stableNo action needed
0.1 – 0.2Warning — minor shiftInvestigate feature distribution
> 0.2Critical — significant driftTrigger retraining pipeline
Section 9 · S6

RLHF Feedback Loop

A React + FastAPI feedback UI with thumbs up/down and star rating, preference dataset builder (JSONL for OpenAI fine-tuning), and automated retraining triggers via GitHub Actions webhooks.

Retraining Trigger Logic

7-day rolling window evaluated nightly. If avg_rating < 3.0 → CRITICAL drift in offer quality → retrain propensity model. If thumbs_down_rate > 0.40 → CRITICAL copy degradation → run eval harness and force prompt rollback if eval < 0.75.

Preference Dataset Format

// feedback_loop/data/preference_dataset.jsonl
{
  "prompt": "Customer tier: Gold. Offer: Free cookie...",
  "chosen": "Treat yourself — enjoy a free cookie on us! Redeem in-app.",
  "rejected": "Get cookie. Use app. Today only."
}

Technical Foundation Required for Level 4

Four foundations must all be strong simultaneously: (1) Reinforcement learning / bandit systems, (2) Causal inference — not just correlation, (3) Strict guardrail enforcement, (4) Business-level explainability. If any one is absent, autonomous operation becomes untrustworthy.
